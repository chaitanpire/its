Deploy Your App to the Host Server

A fast classroom walkthrough for tomorrow: join Tailscale, log in with SSH, upload the project, start the app on the assigned port, and send the merge payload back to the host system.

  • Server name: la-et
  • Tailscale IP: 100.90.57.38
  • Every team already has its own Windows account, folder, and port
  • This deck is designed to be clicked slide by slide using Next and Back
Big Picture 02 / 18

The Flow in One Minute

Say this before opening any terminal.

  • Step 1: Team joins the host Tailscale network with its Gmail.
  • Step 2: Team logs into the server with the team username and password.
  • Step 3: Team uploads its code from the local machine.
  • Step 4: Team moves into its own server folder and extracts the project.
  • Step 5: Team starts the app using the correct command for its framework.
  • Step 6: Team opens the live link and sends session-end data to the merge server.
Important 03 / 18

Tailscale Login Is Not SSH Login

This is the first thing to explain clearly.

Tailscale

  • Uses Gmail or approved email
  • Only gives network access to the host machine
  • Lets teams connect from different internet networks

SSH

  • Uses team username like team7
  • Uses the password given by the host team
  • Actually opens the server machine for deployment work
Step 1 04 / 18

Join Tailscale and Verify Access

These commands run on the team's own laptop after Tailscale is installed.

Verify network access
tailscale ip -4
tailscale status
  • If the device is not visible in the host network, SSH will fail.
  • This is where teams use Gmail, not the team account.
Step 2 05 / 18

Log Into the Server

Example below uses Team 7. Every team replaces only its own team number.

SSH command
ssh team7@100.90.57.38
  • Nothing appears when typing the SSH password. That is normal.
  • After login, the real working folder pattern is C:\classroom_server\teamN\apps\project
Step 3 06 / 18

Upload the ZIP From the Local Machine

Do not try to use your laptop file path inside the remote SSH shell.

Windows laptop

scp "C:\Users\YourName\Downloads\project.zip" team7@100.90.57.38:

Mac laptop

scp "/Users/yourname/Downloads/project.zip" team7@100.90.57.38:
  • This sends the ZIP into the team user's home directory first.
  • That makes the upload more reliable than targeting a deep Windows path directly.
Step 3A 07 / 18

Alternative: Clone the Project Directly

If your code is already on GitHub, this is often cleaner than uploading a ZIP.

Inside the server after SSH login
cd C:\classroom_server\team7\apps\project
git clone https://github.com/your-team/your-repo.git .

# private repo option
git clone https://YOUR_TOKEN@github.com/your-team/your-repo.git .
  • Use this only inside your assigned team folder.
  • Do not use both ZIP upload and git clone in the same folder unless you know exactly what you are replacing.
  • If the folder is not empty, clean it first or clone into a fresh project folder.
Step 4 08 / 18

Move and Extract in the Correct Team Folder

Now switch to the SSH window and unpack inside the assigned deployment folder.

Inside the server
Move-Item "$HOME\project.zip" "C:\classroom_server\team7\apps\project\"
cd C:\classroom_server\team7\apps\project
Expand-Archive -LiteralPath .\project.zip -DestinationPath . -Force
  • Correct folder: C:\classroom_server\team7\apps\project
  • Wrong folder: C:\Users\classroom_server\team7\apps\project
Attached Doc 09 / 18

Windows Team App Start Commands

This slide comes directly from the attached document. Use these commands only after the team has connected through SSH and moved into its own project folder on the server.

  • Replace only the port number such as 8101, 8102, or 8107.
  • Every app must bind to 0.0.0.0 so it is reachable from another device.

Doc Slide 1 10 / 18

Quick Start Commands for Python-Based Apps

Use these exactly from the team project folder on the server.

FastAPI

Start-Process ".\.venv\Scripts\python.exe" -ArgumentList "-m","uvicorn","app.main:app","--host","0.0.0.0","--port","8101"

Flask

Start-Process ".\.venv\Scripts\python.exe" -ArgumentList "-m","flask","run","--host=0.0.0.0","--port=8101"

Django

Start-Process ".\.venv\Scripts\python.exe" -ArgumentList "manage.py","runserver","0.0.0.0:8101"

Streamlit

Start-Process ".\.venv\Scripts\streamlit.exe" -ArgumentList "run","app.py","--server.address","0.0.0.0","--server.port","8101"
Doc Slide 2 11 / 18

Quick Start Commands for JS and Simple Apps

These are the rest of the commands from the attached start-command sheet.

Node / Express

Start-Process "cmd.exe" -ArgumentList "/c","set HOST=0.0.0.0&& set PORT=8101&& npm start"

React / Vite

Start-Process "cmd.exe" -ArgumentList "/c","set HOST=0.0.0.0&& set PORT=8101&& npm run dev -- --host 0.0.0.0 --port 8101"

Next.js

Start-Process "cmd.exe" -ArgumentList "/c","set HOSTNAME=0.0.0.0&& set PORT=8101&& npm run dev -- -H 0.0.0.0 -p 8101"

Plain Python app.py

Start-Process ".\.venv\Scripts\python.exe" -ArgumentList "app.py"
  • For another team, replace every 8101 with that team's assigned port.
  • If the entry point is different, replace app.main:app, manage.py, app.py, or npm script accordingly.
Step 5 12 / 18

Open the Live Link

After the app starts, the browser should use the Tailscale IP and the assigned team port.

Browser check
http://100.90.57.38:8107
http://100.90.57.38:8107/health
  • Do not open 0.0.0.0 in a browser. That is only the bind address.
  • Teams do not need the same Wi-Fi if they are on the same Tailscale network.
Important Warning 13 / 18

Use the URL for Testing, Do Not Hardcode It

The IP and link we give today are deployment values, not values that should be frozen inside source code.

  • Do not hardcode localhost, 127.0.0.1, 100.90.57.38, or fixed test ports inside frontend or backend code.
  • Use the given link only for testing and current deployment access.
  • If your app needs an API base URL, keep it configurable through environment variables or one config file.
  • Relative paths are safer than fixed full URLs wherever possible.
Step 6 14 / 18

Validate the Payload Before POST

These checks prevent avoidable 422 errors and keep the merge data trustworthy.

  • correct_answers + wrong_answers must be less than or equal to questions_attempted
  • questions_attempted must be less than or equal to total_questions
  • hints_used must be less than or equal to total_hints_embedded
  • topic_completion_ratio must stay between 0 and 1
  • session_status must be only completed or exited_midway
  • chapter_id must match your assigned chapter
  • timestamp should be a valid ISO datetime string
Step 7 15 / 18

Send the Merge Payload at Session End

The deployed chapter app should call the host merge endpoint when the learner finishes or exits.

Sample payload
POST /merge/interactions
{
  "student_id": "student_204",
  "session_id": "team7_demo_001",
  "chapter_id": "grade7_triangle_properties",
  "timestamp": "2026-04-07T11:15:00Z",
  "session_status": "completed",
  "correct_answers": 8,
  "wrong_answers": 2,
  "questions_attempted": 10,
  "total_questions": 10,
  "hints_used": 1,
  "total_hints_embedded": 4,
  "retry_count": 1,
  "time_spent_seconds": 780,
  "topic_completion_ratio": 1.0
}
  • This is what the host merge system reads to decide next topic or support topic.
  • The team app owns the payload values. The host app owns the routing logic.
Common Mistakes 16 / 18

What Will Go Wrong If We Do Not Say It Out Loud

Call these out clearly while presenting.

  • Using the Gmail inside SSH instead of the team username
  • Trying to use a laptop file path from inside the remote SSH terminal
  • Uploading to the wrong Windows path
  • Opening 0.0.0.0 in the browser
  • Thinking the password is not being typed because nothing appears on screen
  • Forgetting to change the port for the correct team
Final Slide 17 / 18

What Every Team Must Remember

If they remember these five points, the deployment part will go smoothly.

  • Use Gmail for Tailscale and team credentials for SSH.
  • Upload from the local terminal with scp.
  • Work only in the assigned team folder.
  • Start the app on 0.0.0.0 and the assigned port.
  • Send session-end data to POST /merge/interactions.
API Final Slide 18 / 18

GET, POST, and Merge APIs We Will Use Tomorrow

This is the exact API picture teams need: what they call, what they send, and what the host returns.

GET APIs

  • GET http://100.90.57.38:8200/health
  • GET http://100.90.57.38:8200/merge/chapters
  • GET http://100.90.57.38:8200/merge/recommendations/student_204
  • GET http://100.90.57.38:8200/merge/next-chapter/student_204

POST APIs

  • POST http://100.90.57.38:8200/merge/interactions
  • POST http://100.90.57.38:8200/demo/student-session
  • GET tells us what the host knows.
  • POST sends the session data from the team app back to the host.

GET /health

{
  "status": "ok"
}

GET /merge/next-chapter/{student_id}

{
  "decision_type": "advance",
  "predicted_next_chapter_id": "grade7_perimeter_area",
  "predicted_next_chapter_name": "Perimeter and Area"
}

GET /merge/recommendations/{student_id}

{
  "needs_support": true,
  "recommendations": ["grade7_rational_numbers"],
  "weak_subtopics": ["signed operations"]
}

POST /merge/interactions

{
  "status": "created",
  "student_id": "student_204",
  "chapter_id": "grade7_triangle_properties"
}
How teams usually send the merge payload
curl -X POST http://100.90.57.38:8200/merge/interactions ^
  -H "Content-Type: application/json" ^
  -d "{
    \"student_id\": \"student_204\",
    \"session_id\": \"team7_demo_001\",
    \"chapter_id\": \"grade7_triangle_properties\",
    \"timestamp\": \"2026-04-08T11:15:00Z\",
    \"session_status\": \"completed\",
    \"correct_answers\": 8,
    \"wrong_answers\": 2,
    \"questions_attempted\": 10,
    \"total_questions\": 10,
    \"hints_used\": 1,
    \"total_hints_embedded\": 4,
    \"retry_count\": 1,
    \"time_spent_seconds\": 780,
    \"topic_completion_ratio\": 1.0
  }"
  • GET is for checking status and reading routing results.
  • POST /merge/interactions is the main handoff from every deployed chapter app to the host.
  • After POST succeeds, the host can answer recommendations and next-chapter requests for that student.
